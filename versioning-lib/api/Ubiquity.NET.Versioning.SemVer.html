Remarks

Officially a SemVer value does NOT limit the size of the numeric portions of a version so this implementation uses BigInteger values for each component. This allows all possible values.

In practical terms any such component will likely "down convert" to an integer. If a version component in the real world exceeds the size of an integer, then there is probably something wrong with how the versioning is maintained.

The comparison/ordering of this type is based on the rules for Semantic Versioning with the extension of AlphaNumericOrdering. Sadly, the SemVer spec is silent on the point of case comparisons and different major component repositories have chosen different interpretations of the spec as a result. Thus any consumer must explicitly decide which comparison a version expects and that is part of the version.

important

Due to all the confusion and ambiguity surrounding the use of case sensitivity in ordering of semantic versions it is STRONGLY recommended that ALL use of [C]SemVer[-CI] value strings always uses a single case form. (That is 'ALL UPPERCASE`, or 'all lower case' so that the versions are ordered consistently no matter what a particular repository uses. This is especially true of CSemVer[-CI] variants when used in a repository that supports case sensitive SemVer as the spec for CSemVer[-CI] explicitly calls out a case insensitive comparison.

note

Technically, the SemVer spec states that alphanumeric Identifiers are ordered lexicographically, which would make them case sensitive. However, since MAJOR framework repositories have chosen to use each approach the real world of ambiguity, sadly, wins.
